# CI/CD Auto-Publish Design

## Overview

Validate the GitHub Actions auto-publish pipeline for the `mini-play` Flutter project before developing actual games. Uses the Flutter default Counter App as the test payload.

## Prerequisites

- Scaffold a Flutter project in the repo root via `flutter create .` (the repo currently only has a README.md)
- Enable GitHub Pages in repo settings: Settings > Pages > Source: "Deploy from a branch" > Branch: `gh-pages` / `/ (root)`

## Goals

- Verify Web deployment to GitHub Pages works end-to-end
- Verify Android debug APK build and upload to GitHub Releases
- Establish CI/CD infrastructure that supports future game development
- Keep it simple — minimal config, easy to extend later

## Target Platforms

| Platform | Output | Destination |
|----------|--------|-------------|
| Web | Static site | GitHub Pages (`https://cecil-su.github.io/mini-play/`) |
| Android | Debug APK | GitHub Releases (tag-triggered only) |

```
mini-play/

│       └── ci.yml              # Single CI/CD workflow
├── .fvmrc                      # Flutter version lock
├── lib/
│   └── main.dart               # Flutter default Counter App

├── pubspec.yaml

```

Single Flutter project. All future games will live within this app (game list on the home page, tap to enter each game).

## Flutter Version Management

- `.fvmrc` in project root pins the Flutter version (e.g., `3.27.4`)
- CI dynamically reads the version from `.fvmrc` and passes it to `subosito/flutter-action` via a setup step
- Locally, the user uses `fvm` (installed via scoop)

## CI/CD Workflow Design

### Trigger Conditions

- **Push tag `v*`** (e.g., `v0.1.0`) — full build + deploy + release
- **`workflow_dispatch`** (manual) — build Web + deploy Web only (no Android build, no GitHub Release)

### Permissions

Workflow-level permissions required:

```yaml
permissions:
  contents: write    # Push to gh-pages branch + create GitHub Releases
```

### Job Structure

**Tag trigger:**
```
┌─────────────┐    ┌───────────────┐
│  build-web  │    │ build-android │    ← Run in parallel
│  (ubuntu)   │    │   (ubuntu)    │
└──────┬──────┘    └───────┬───────┘
       │                   │
       └─────────┬─────────┘
                 │
          ┌──────▼──────┐
          │   deploy    │    ← Waits for both builds
          │  (ubuntu)   │
          └─────────────┘
```

**Manual trigger:**
```
┌─────────────┐
│  build-web  │
│  (ubuntu)   │
└──────┬──────┘
       │
┌──────▼──────┐
│   deploy    │    ← Web only
│  (ubuntu)   │
└─────────────┘
```

### build-web Job

1. Checkout code
2. Install Flutter via `subosito/flutter-action@v2` (version read from `.fvmrc`)
3. Run `flutter pub get`
4. Run `flutter build web --release --base-href "/mini-play/"`
5. Upload `build/web` as artifact named `web-build` (via `actions/upload-artifact@v4`)

### build-android Job

**Condition**: only runs on tag triggers (`if: startsWith(github.ref, 'refs/tags/v')`)

1. Checkout code
2. Install Java 17 (`actions/setup-java@v4`)
3. Install Flutter via `subosito/flutter-action@v2` (version read from `.fvmrc`)
4. Run `flutter pub get`
5. Run `flutter build apk --debug`
6. Rename APK to `mini-play-debug.apk`
7. Upload APK as artifact named `android-apk` (via `actions/upload-artifact@v4`)

### deploy Job

**Depends on**: `build-web` (always) + `build-android` (on tag triggers)

1. Download artifact `web-build`
2. Deploy Web to GitHub Pages via `peaceiris/actions-gh-pages@v3` (publishes to `gh-pages` branch, uses `GITHUB_TOKEN`)
3. **(Tag only)** Download artifact `android-apk`
4. **(Tag only)** Create GitHub Release via `softprops/action-gh-release@v2`:
   - Tag name from `github.ref_name` (e.g., `v0.1.0`)
   - Release title: tag name
   - Release body: auto-generated by GitHub (`generate_release_notes: true`)
   - Attach `mini-play-debug.apk`

## Release Strategy

| Trigger | Web Deploy | Android Build | GitHub Release (APK) |
|---------|-----------|---------------|---------------------|
| Push tag `v*` | Yes | Yes | Yes |
| Manual `workflow_dispatch` | Yes | No | No |

## GitHub Pages Configuration

- Published from `gh-pages` branch
- `base-href` set to `/mini-play/` to match repository name
- Access URL: `https://cecil-su.github.io/mini-play/`
- Must enable Pages in repo settings (Settings > Pages > Source: `gh-pages` branch)

## Future Extension: Release Signing

When ready to switch from debug to release APK:

1. Generate a keystore file
2. Add GitHub Secrets: `KEYSTORE_BASE64`, `KEY_ALIAS`, `KEY_PASSWORD`, `STORE_PASSWORD`
3. In `build-android` job: decode keystore from secret, configure `key.properties`
4. Change `flutter build apk --debug` to `flutter build apk --release`
5. Add `android/key.properties` to `.gitignore`

No structural changes needed — just parameter updates in the existing workflow.
